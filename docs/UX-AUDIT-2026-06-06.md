## P1 — 最優先（日常で踏む / 正確性 / 中核用語）

### UX-P1-01 [ジェスチャ] カードに `touch-action` 指定が無く、200ms 長押しドラッグの掴みが不安定
- 現状: `src/components/TaskCardView.tsx:34-38` のカード root に `{...drag.listeners}`（`TaskCard.tsx:35`）を spread するが `touch-action` 指定が無い（`index.css` にも無し）。dnd-kit TouchSensor（`Board.tsx:55` `delay:200, tolerance:8`）は `touchmove` の preventDefault で掴むが、`touch-action:auto` だと最初の微動で縦/横スクローラに取られ、長押しが空振り／指の下で盤面が動く。
- なぜ: カードは「縦スクロール列の中の横スクロール行の中」にあり、毎日触る中核操作。「ドラッグが不安定」の最大要因。
- 修正: ドラッグ起点に `touch-action:none` を効かせる。全カードに付けるとスクロール開始が殺せるので、**カードに専用ドラッグハンドル（≡）を設け、ハンドルだけ `touch-action:none`**（ラベル並べ替えで採った方式と同じ）にするのが綺麗。
- 工数: M

### UX-P1-02 [状態] バックグラウンド/手動 refresh の失敗が無音 → 古い盤面のまま気づけない
- 現状: `BoardContext.tsx:121-132` の `refresh()` は失敗を `setError` するが、Board の表示は `board.error && board.tasks.length === 0`（`Board.tsx:73`）でガードされ、**既にカードがある状態の失敗は一切描画されない**。focus/visibilitychange の自動 refresh（`:139-151`）も手動更新ボタンも無音で失敗（スピナーが止まるだけ）。
- なぜ: iPhone は前面復帰のたび裏で refresh。圏外復帰直後などに失敗すると「最新化された」と誤認して古いデータを操作する。信頼性に直結。
- 修正: データがある状態での refresh 失敗は控えめな toast（`移動に失敗` と同形）かヘッダ下の細バナーで通知。最低限、手動更新ボタン経由の失敗は toast。
- 工数: M

### UX-P1-03 [不具合/自己回帰] 列の縦スクロール復元が「空マウント one-shot」で、データ後着の列が先頭に戻る
- 現状: `StatusColumn.tsx:37-41`（および `CompletedColumn.tsx:17-21`）の `useLayoutEffect` は初回 1 回で `didRestore.current=true` を立てる。列が空（データ未到着）でマウントすると、空状態で復元（scrollTop=0）してフラグを消費し、**タスク到着後に保存済みスクロールが二度と当たらない**。Board 横位置（`Board.tsx:66-70`）は「当たるまで毎レンダー再試行」なのに、列縦位置だけ one-shot。
- なぜ: 先日入れた「縦スクロール保持」が、詳細から戻った直後にデータが 1 tick 遅れて来るケースで効かない＝機能が約束を破る。
- 修正: Board と同じく「復元できる中身がある時だけ確定」にする（`scrollHeight>clientHeight` を条件、または `tasks.length` を依存に再試行）。**UX-P3 の『layoutEffect 毎レンダー実行』と同時に、データ依存 effect へ作り替えると一石二鳥**。
- 工数: S

### UX-P1-04 [フォーム] 保存ボタンが `dirty` を無視 → 無変更でも保存・「保存しました」・無駄な API 書込
- 現状: `TaskDetail.tsx:220` の保存ボタン `disabled={saving || !title.trim()}` は `dirty`（`:92` で算出済み）を見ない。何も変えずに保存すると PATCH が走り（`:131`）成功 toast が出て、GitHub の updatedAt も動く・レート消費。
- なぜ: 「変えていないのに保存しました」は誤解を生み、無駄な書き込みが起きる。`dirty` を計算しているのに使っていない明確なバグ。
- 修正: `disabled` に `|| !dirty` を追加（未変更時は編集を閉じるだけでも可）。
- 工数: S

### UX-P1-05 [フォーム/IME] 日本語入力の要：`enterKeyHint`/`inputMode` 不在＋Enter 送信不在＋IME 変換確定の誤送信ガード不在
- 現状: タイトル/本文/コメントの入力に `enterKeyHint`/`inputMode` 指定なし（`Input.tsx:6-13`/`:25`、`TaskDetail.tsx:208`、`QuickAddSheet.tsx:70-76`、`CommentList.tsx:44-48`）。Enter 送信は QuickAdd の `<form>` のみ（`:67`）で、編集フォームは素の `<div>`、コメントは Enter=改行。さらに `isComposing`/`onCompositionEnd` のハンドリングがコード全体でゼロ。
- なぜ: 毎日日本語でタスク起票するアプリで、iOS キーキャップが汎用「改行」のまま／片手で確定・送信できない。Enter 送信を足すなら、**変換確定 Enter の誤送信ガード（`e.nativeEvent.isComposing` 判定）が前提として必須**（「かいぎ」を漢字確定する瞬間に未確定で起票される事故）。
- 修正: 用途別に `enterKeyHint`（タイトル=`done`/コメント=`send`）を付与。タイトルは `<form onSubmit>` 化、コメントは Cmd/Ctrl+Enter 送信。全 Enter 送信箇所に `if (e.nativeEvent.isComposing) return` を共通化。
- 工数: M

### UX-P1-06 [コピー] 「完了」を表す語が 完了済み／クローズ済み／Done と画面ごとにバラバラ
- 現状: 完了 issue のバッジは `クローズ済み`（`TaskDetail.tsx:259-260`）、ボード右端列は `完了済み`（`CompletedColumn.tsx:26`）、トグルは `完了したタスクも表示`（`Settings.tsx:191`）、ボタンは `完了にする`（`TaskDetail.tsx:292`）、status は `Done`（`status.ts:19`）。
- なぜ: ネイティブ日本語ユーザーが「クローズ済み」と「完了済み」を別概念と誤読。同一状態の表記ゆれは用語の単一情報源の崩れ。
- 修正: 完了状態は **「完了済み」に統一**（`TaskDetail.tsx:260` の `クローズ済み`→`完了済み`）。GitHub 用語を出すなら「完了（クローズ済み）」と全画面で統一。
- 工数: S

## P2 — 体感を明確に上げる改善

### ジェスチャ・ドラッグ・スクロール

**UX-P2-01 [ドラッグ] 列間ドラッグの auto-scroll が未調整で、画面外の列/列の底へ運べない**

**UX-P2-04 [スナップ] `snap-proximity` × `72vw` で列が常に半端に切れ、復元スクロールと競合**
